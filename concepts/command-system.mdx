Slash commands are the primary way users interact with Claude Code's built-in features. The command system is responsible for loading, filtering, and dispatching commands from a variety of sources.

## Command types

Every command has a `type` field that determines how it executes. The three types are defined in `src/types/command.ts`:

<CardGroup cols={3}>
  <Card title="local" icon="bolt">
    Runs a TypeScript function directly. Returns a `LocalCommandResult` — either `text`, a `compact` result, or `skip`. Examples: `/clear`, `/cost`, `/compact`.
  </Card>
  <Card title="local-jsx" icon="react">
    Renders a React/Ink UI component in the terminal. Calls `onDone()` when finished. Examples: `/mcp`, `/config`, `/theme`.
  </Card>
  <Card title="prompt" icon="message">
    Expands to a block of text that is sent to the model. Used for skills and slash-command shortcuts. Examples: custom skills in `.claude/skills/`.
  </Card>
</CardGroup>

```typescript
// From src/types/command.ts

type LocalCommand = {
  type: 'local'
  supportsNonInteractive: boolean
  load: () => Promise<LocalCommandModule>
}

type LocalJSXCommand = {
  type: 'local-jsx'
  load: () => Promise<LocalJSXCommandModule>
}

type PromptCommand = {
  type: 'prompt'
  progressMessage: string
  contentLength: number
  getPromptForCommand(args: string, context: ToolUseContext): Promise<ContentBlockParam[]>
}
```

Both `local` and `local-jsx` commands use lazy `load()` functions so their implementation modules are never imported until the command is actually invoked.

## Command sources

The `source` field on a command tells callers and the UI where the command came from:

| Source | Description |
|---|---|
| `builtin` | Shipped inside Claude Code itself (`src/commands/`) |
| `bundled` | Compiled into the bundle at build time (`getBundledSkills()`) |
| `skills` | Loaded from a `.claude/skills/` directory on disk |
| `plugin` | Provided by a user-installed plugin |
| `mcp` | Provided by a connected MCP server |
| `commands_DEPRECATED` | Legacy `.claude/commands/` directory (still supported) |

The `loadedFrom` field is a parallel discriminant that tracks the physical origin of a `prompt` command:

```typescript
loadedFrom?:
  | 'commands_DEPRECATED'
  | 'skills'
  | 'plugin'
  | 'managed'
  | 'bundled'
  | 'mcp'
```

## getCommands() flow

`getCommands(cwd)` is the public API for retrieving the list of commands available to the current user. It has three stages:

<Steps>
  <Step title="loadAllCommands(cwd)">
    Memoized by `cwd`. Loads all command sources in parallel (bundled skills, plugin skills, skill directories, workflow commands, plugin commands, built-in commands) and concatenates them.
  </Step>
  <Step title="getDynamicSkills()">
    Returns skills that were discovered during the session (e.g. when the model touches a file that matches a skill's `paths` glob). These are merged in, deduped, and inserted before the built-in commands.
  </Step>
  <Step title="Filter by availability and enabled state">
    Each command is tested against `meetsAvailabilityRequirement()` and `isCommandEnabled()`. Commands that fail either check are excluded from the result.
  </Step>
</Steps>

```typescript
export async function getCommands(cwd: string): Promise<Command[]> {
  const allCommands = await loadAllCommands(cwd)
  const dynamicSkills = getDynamicSkills()

  const baseCommands = allCommands.filter(
    _ => meetsAvailabilityRequirement(_) && isCommandEnabled(_),
  )

  if (dynamicSkills.length === 0) {
    return baseCommands
  }

  // Insert dynamic skills after plugin skills but before built-in commands
  const builtInNames = new Set(COMMANDS().map(c => c.name))
  const insertIndex = baseCommands.findIndex(c => builtInNames.has(c.name))
  // ...
}
```

### meetsAvailabilityRequirement()

Commands can declare an `availability` array restricting them to specific auth types:

```typescript
export function meetsAvailabilityRequirement(cmd: Command): boolean {
  if (!cmd.availability) return true
  for (const a of cmd.availability) {
    switch (a) {
      case 'claude-ai':
        if (isClaudeAISubscriber()) return true
        break
      case 'console':
        if (!isClaudeAISubscriber() && !isUsing3PServices() && isFirstPartyAnthropicBaseUrl())
          return true
        break
    }
  }
  return false
}
```

This check runs fresh on every `getCommands()` call because auth state can change mid-session (e.g. after `/login`).

### isCommandEnabled()

```typescript
export function isCommandEnabled(cmd: CommandBase): boolean {
  return cmd.isEnabled?.() ?? true
}
```

Commands without `isEnabled` are always enabled. Commands with `isEnabled` use it to gate on feature flags, environment variables, or other runtime conditions.

## Skill commands

Two filtered views of `getCommands()` are used to power the model's skill invocation:

### getSkillToolCommands()

Returns all `prompt`-type commands that the model can invoke directly. This includes skills from `/skills/` directories, bundled skills, and legacy `/commands/` entries. Plugin and MCP commands are included only if they have an explicit description or `whenToUse`.

```typescript
export const getSkillToolCommands = memoize(
  async (cwd: string): Promise<Command[]> => {
    const allCommands = await getCommands(cwd)
    return allCommands.filter(
      cmd =>
        cmd.type === 'prompt' &&
        !cmd.disableModelInvocation &&
        cmd.source !== 'builtin' &&
        (cmd.loadedFrom === 'bundled' ||
          cmd.loadedFrom === 'skills' ||
          cmd.loadedFrom === 'commands_DEPRECATED' ||
          cmd.hasUserSpecifiedDescription ||
          cmd.whenToUse),
    )
  },
)
```

### getSlashCommandToolSkills()

Returns `prompt`-type commands that represent skills (as opposed to plain slash-command shortcuts). A skill must have `loadedFrom` of `skills`, `plugin`, or `bundled`, or have `disableModelInvocation` set.

```typescript
export const getSlashCommandToolSkills = memoize(
  async (cwd: string): Promise<Command[]> => {
    const allCommands = await getCommands(cwd)
    return allCommands.filter(
      cmd =>
        cmd.type === 'prompt' &&
        cmd.source !== 'builtin' &&
        (cmd.hasUserSpecifiedDescription || cmd.whenToUse) &&
        (cmd.loadedFrom === 'skills' ||
          cmd.loadedFrom === 'plugin' ||
          cmd.loadedFrom === 'bundled' ||
          cmd.disableModelInvocation),
    )
  },
)
```

## MCP skill commands

MCP servers can provide `prompt`-type commands. These are filtered separately from `getCommands()` because they live in `AppState.mcp.commands` rather than the normal command registry:

```typescript
export function getMcpSkillCommands(
  mcpCommands: readonly Command[],
): readonly Command[] {
  if (feature('MCP_SKILLS')) {
    return mcpCommands.filter(
      cmd =>
        cmd.type === 'prompt' &&
        cmd.loadedFrom === 'mcp' &&
        !cmd.disableModelInvocation,
    )
  }
  return []
}
```

## Dynamic skills

`getDynamicSkills()` (from `src/skills/loadSkillsDir.ts`) returns skills discovered at runtime — typically when the model reads a file whose path matches a skill's `paths` glob pattern. These are injected into `getCommands()` automatically and deduplicated against the base command list.

## Finding and getting commands

Three utility functions look up commands by name:

```typescript
// Returns undefined if not found
export function findCommand(
  commandName: string,
  commands: Command[],
): Command | undefined {
  return commands.find(
    _ =>
      _.name === commandName ||
      getCommandName(_) === commandName ||
      _.aliases?.includes(commandName),
  )
}

// Returns true if found
export function hasCommand(commandName: string, commands: Command[]): boolean {
  return findCommand(commandName, commands) !== undefined
}

// Returns the command or throws ReferenceError with a list of available names
export function getCommand(commandName: string, commands: Command[]): Command {
  const command = findCommand(commandName, commands)
  if (!command) {
    throw ReferenceError(`Command ${commandName} not found. Available commands: ...`)
  }
  return command
}
```

All three match on `cmd.name`, `cmd.userFacingName()`, and any entries in `cmd.aliases`.

## Remote and bridge safety

Two sets restrict which commands are available in remote contexts:

### REMOTE_SAFE_COMMANDS

Commands that are safe to show before the CCR (Claude Code Remote) connection is established, because they only affect local TUI state:

```typescript
export const REMOTE_SAFE_COMMANDS: Set<Command> = new Set([
  session,   // Shows QR code / URL for remote session
  exit,      // Exit the TUI
  clear,     // Clear screen
  help,      // Show help
  theme,     // Change terminal theme
  color,     // Change agent color
  vim,       // Toggle vim mode
  cost,      // Show session cost
  usage,     // Show usage info
  copy,      // Copy last message
  btw,       // Quick note
  feedback,  // Send feedback
  plan,      // Plan mode toggle
  keybindings,
  statusline,
  stickers,
  mobile,
])
```

### BRIDGE_SAFE_COMMANDS

`local`-type commands that are safe to execute when received over the Remote Control bridge (from a mobile or web client). `local-jsx` commands are always blocked (they render terminal UI); `prompt` commands are always allowed (they expand to text):

```typescript
export const BRIDGE_SAFE_COMMANDS: Set<Command> = new Set(
  [
    compact,      // Shrink context
    clear,        // Wipe transcript
    cost,         // Show session cost
    summary,      // Summarize conversation
    releaseNotes, // Show changelog
    files,        // List tracked files
  ].filter((c): c is Command => c !== null),
)

export function isBridgeSafeCommand(cmd: Command): boolean {
  if (cmd.type === 'local-jsx') return false
  if (cmd.type === 'prompt') return true
  return BRIDGE_SAFE_COMMANDS.has(cmd)
}
```

## UI display formatting

`formatDescriptionWithSource()` annotates a command's description with its origin, used in typeahead and help screens:

```typescript
export function formatDescriptionWithSource(cmd: Command): string {
  if (cmd.type !== 'prompt') return cmd.description
  if (cmd.kind === 'workflow') return `${cmd.description} (workflow)`
  if (cmd.source === 'plugin') {
    const pluginName = cmd.pluginInfo?.pluginManifest.name
    if (pluginName) return `(${pluginName}) ${cmd.description}`
    return `${cmd.description} (plugin)`
  }
  if (cmd.source === 'builtin' || cmd.source === 'mcp') return cmd.description
  if (cmd.source === 'bundled') return `${cmd.description} (bundled)`
  return `${cmd.description} (${getSettingSourceName(cmd.source)})`
}
```

For model-facing prompts (such as `SkillTool`), use `cmd.description` directly — source annotations are for human display only.

## Cache invalidation

Two invalidation functions are provided:

```typescript
// Clears only the memoization caches for commands.
// Use when dynamic skills are added mid-session.
export function clearCommandMemoizationCaches(): void {
  loadAllCommands.cache?.clear?.()
  getSkillToolCommands.cache?.clear?.()
  getSlashCommandToolSkills.cache?.clear?.()
  clearSkillIndexCache?.()
}

// Full cache clear: also clears plugin caches and skill directory caches.
// Use when plugins or skills change on disk.
export function clearCommandsCache(): void {
  clearCommandMemoizationCaches()
  clearPluginCommandCache()
  clearPluginSkillsCache()
  clearSkillCaches()
}
```

<Note>
`clearCommandMemoizationCaches()` only invalidates the inner caches. If a skill search index is built on top of those caches, it must be cleared separately — which is why `clearSkillIndexCache?.()` is called explicitly.
</Note>
